# LAN And Multiplayer Feature Plan

Last reviewed: 2026-05-27

This plan tracks explicit LAN support and the multiplayer feature path. The project currently has a secure local-default engine: Docker publishes the engine only on host loopback, React pairs with a local engine URL, and the desktop pairing token gates engine HTTP and Socket.IO access.

The goal is to add LAN multiplayer without accidentally turning every running desktop engine into a network service.

## Product Goal

Let a host run the Pixelated engine on their machine and invite other devices on the same LAN to join a game session.

Expected first version:

- The host explicitly enables LAN mode in the desktop app.
- The desktop app shows the LAN engine URL, pairing token, and warning copy.
- Another device on the same LAN opens the hosted React app, enters the LAN engine URL and token, and can join a session.
- The host can disable LAN mode and rotate the token.
- Loopback-only mode remains the default.

## Non-Goals For The First LAN Slice

- Internet multiplayer through NAT traversal.
- Hosted engine fleet scheduling.
- Public engine discovery.
- Multiple independent emulator instances in one local engine.
- Replacing the backend API with peer-to-peer auth.

## Current Constraints

- `apps/desktop/main.js` currently starts Docker with `-p 127.0.0.1:8080:8080`.
- The engine process listens on `0.0.0.0:8080` inside the container, but Docker only exposes it on host loopback.
- React defaults to `http://localhost:8080`, with an engine URL override stored in browser local storage.
- The pairing token is generated per desktop engine run and is required for local engine HTTP routes and Socket.IO.
- The backend stores local pairing intent metadata, but never stores the desktop pairing token.
- The engine currently supports one active RetroArch/camera pair at a time.
- Socket.IO rooms scope signaling by session id, but the runtime process manager is still one-active-game-oriented.
- Local Vault user identity is still trusted from `X-User-Id` or socket payload; possession of the engine token is the real local boundary today.

## Security Principles

- LAN mode is opt-in and visible in the desktop UI.
- Loopback-only stays the default for normal single-player local use.
- Enabling or disabling LAN mode rotates the desktop pairing token.
- The UI explains that anyone on the LAN with the token can attempt to use the engine.
- CORS stays explicit: hosted Vercel, local dev origins, and no wildcard origin.
- The backend may store engine URL metadata, but never stores the desktop pairing token.
- Tokens and invite links should be short-lived where possible.
- Wrong token, disabled LAN mode, and unexpected origins must fail closed.

## Proposed Architecture

### LAN Engine Exposure

Add a desktop-level engine exposure mode:

- `local`: publish Docker as `127.0.0.1:8080:8080`.
- `lan`: publish Docker as `0.0.0.0:8080:8080`, or preferably a selected host LAN IP when Docker/platform support is reliable.

The desktop app should discover host LAN IPv4 addresses and display the likely URL:

`http://<lan-ip>:8080`

The engine should also expose non-secret mode metadata through `/health`, such as:

- `exposureMode`
- `tokenRequired`
- `activeSessionId`
- optionally `advertisedUrls`

### Pairing And Invite Model

Initial LAN pairing can reuse the current engine URL plus pairing token flow:

1. Host enables LAN mode in desktop.
2. Desktop rotates token and shows LAN URL plus token.
3. Guest opens hosted React.
4. Guest enters `http://<host-lan-ip>:8080` and token.
5. React checks `/health`, then uses the token for Socket.IO and local engine REST calls.

Next hardening layer:

- Add short invite codes generated by the desktop app.
- Make invite codes expire after a few minutes.
- Exchange invite code for the current engine token locally, without sending secrets to the backend.
- Let host revoke active invites.

### Multiplayer Session Model

Add explicit participant roles:

- `host`: owns game boot, stop, and session settings.
- `player`: can send input for an assigned player slot.
- `spectator`: receives stream only.
- `camera`: internal sender role used by `camera.py`.

Add player slots:

- Slot 1 defaults to host.
- Slots 2 to 4 can be assigned to LAN guests.
- Each slot has a display name, connection state, and input permissions.

Session state should live in the local engine for LAN mode first. The backend can later mirror non-secret lobby metadata for signed-in users.

### Input Routing

The current input event shape is effectively one-player keyboard input. Multiplayer needs explicit slot routing:

```json
{
  "sessionId": "string",
  "playerIndex": 1,
  "inputType": "keyboard",
  "key": "ArrowRight",
  "action": "keydown"
}
```

Engine responsibilities:

- Verify the socket token.
- Verify the socket is assigned to the requested player slot.
- Reject input for unassigned or spectator sockets.
- Map each player slot to the correct RetroArch input port.

Open technical question: direct `xdotool` injection may only be clean for one keyboard-like controller. Real multiplayer may need RetroArch input config, virtual gamepads, or RetroArch netplay instead of only X11 keyboard injection.

### Media Model

First LAN multiplayer version can keep one shared WebRTC media stream per session:

- Host starts game.
- Browser clients join the same session room.
- Each browser negotiates its own WebRTC connection to the same running `camera.py` sender, or the engine starts one sender per viewer if GStreamer cannot fan out cleanly.

Open technical question: GStreamer `webrtcbin` may need one peer connection per remote browser. The plan should validate whether the current single `camera.py` pipeline can support multiple peers or whether the local engine needs a per-viewer camera bridge.

## Implementation Phases

### Phase 1: Explicit LAN Mode Toggle

Status: planned.

Deliverables:

- Add a desktop LAN mode toggle, default off.
- Rotate the pairing token whenever LAN mode changes.
- Start Docker with loopback publish in local mode.
- Start Docker with LAN publish in LAN mode.
- Display discovered LAN URL(s) in the desktop UI.
- Update `/health` to report exposure mode.
- Update `.context/current-infrastructure.md` and `.context/project-flows.md`.

Acceptance criteria:

- Default engine startup still publishes only `127.0.0.1:8080`.
- LAN mode cannot be enabled silently.
- `http://localhost:8080/health` works in local mode.
- `http://<lan-ip>:8080/health` works from another LAN device only after LAN mode is enabled.
- Disabling LAN mode stops/restarts the container back on loopback and rotates the token.

### Phase 2: LAN Pairing UX

Status: planned.

Deliverables:

- Update `EnginePairingPanel` copy and validation to support LAN URLs intentionally.
- Show whether a saved engine URL is loopback or LAN.
- Add clearer wrong-token and unreachable-host messages.
- Persist backend pairing metadata as current behavior, still without storing the token.

Acceptance criteria:

- A guest browser can pair with a LAN engine URL and token.
- Wrong token is rejected by HTTP and Socket.IO.
- Pairing metadata saved to the backend contains only non-secret URL data.

### Phase 3: Lobby And Roles

Status: planned.

Deliverables:

- Add local engine session/lobby state.
- Add `host`, `player`, and `spectator` roles.
- Add join/leave events.
- Add player slot assignment.
- Let host kick/revoke participants.

Acceptance criteria:

- Host can see connected participants.
- Guests cannot start/stop games unless assigned host permissions.
- Spectators receive stream but cannot send input.

### Phase 4: Multiplayer Input Routing

Status: planned.

Deliverables:

- Extend React input events with `playerIndex`.
- Add engine-side participant/slot validation.
- Add player 2+ key/gamepad mapping strategy.
- Add tests for rejected unauthorized input.

Acceptance criteria:

- Player 1 and Player 2 input routes to distinct emulator controls in a test ROM or RetroArch input diagnostic.
- A spectator cannot control the emulator.
- A guest assigned to slot 2 cannot emit slot 1 input.

### Phase 5: Multi-Viewer WebRTC Validation

Status: planned.

Deliverables:

- Test whether one `camera.py` sender can negotiate multiple browser peers.
- If not, design per-viewer sender processes or a local media fanout strategy.
- Add session cleanup so viewer disconnects do not stop the host game unless the host leaves.

Acceptance criteria:

- Two LAN browsers can receive audio/video from the same game session.
- Disconnecting one guest does not kill the host session.
- Stopping the host session cleans up all viewer signaling/media state.

### Phase 6: Backend Multiplayer Support

Status: planned, after local LAN proof.

Deliverables:

- Add backend lobby records for signed-in users if needed.
- Add optional invite metadata without storing engine tokens.
- Add audit/metrics for multiplayer sessions.
- Keep engine token exchange local.

Acceptance criteria:

- Backend can show recent/active multiplayer metadata without being able to control a user's local engine.
- Secret material remains local to the host and invited browser clients.

## Test Plan

Automated:

- Desktop bind mode command construction.
- Token rotation on exposure mode changes.
- Engine `/health` exposure mode response.
- Socket role and slot authorization.
- Input event validation.

Manual:

- Host machine starts local mode and confirms LAN device cannot reach the engine.
- Host enables LAN mode and confirms LAN device can reach `/health`.
- LAN guest pairs through hosted React using LAN URL plus token.
- Host starts a game and guest receives stream.
- Guest input works only for assigned slot.
- Host disables LAN mode and guest connection fails closed.

## Open Decisions

- Should LAN mode bind to `0.0.0.0` or a selected host interface/IP?
- Should the first multiplayer build use shared keyboard mapping, virtual gamepads, or RetroArch netplay?
- Should invite codes be implemented before any LAN pairing ships publicly?
- Should guest pairing require signed-in users, or should LAN guests be allowed as token-only guests?
- Should the host be able to choose max players and spectator permissions per session?

## Recommended Next Implementation Task

Start with Phase 1: explicit LAN mode toggle in the desktop app.

That is the smallest valuable slice because it preserves the secure default while giving the project a real LAN surface to test before adding lobbies, player slots, and multi-viewer media behavior.
